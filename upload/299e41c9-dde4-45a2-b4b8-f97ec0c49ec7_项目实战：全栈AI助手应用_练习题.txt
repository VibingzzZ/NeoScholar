## 全栈AI助手应用实战练习题

### 一、选择题

**1. 在开发全栈AI助手应用时，为了在用户与AI的对话中保持上下文连贯性，最推荐的做法是什么？**
A. 每次用户提问时，将整个对话历史发送给LLM API。
B. 在数据库中存储用户的会话ID，每次请求时只发送最新的用户消息。
C. 在前端维护一个消息数组，每次请求时将该数组作为上下文发送给后端，由后端拼接后传给LLM API。
D. 让LLM API自动记忆所有历史对话。

**正确答案：C**

**解析：** 选项A虽然能保证上下文，但会消耗大量token，成本高且效率低；选项B完全丢失了上下文；选项D不现实，LLM本身无状态。选项C是标准做法，前端管理会话状态，后端负责组织并发送给LLM，既控制了上下文长度，又保证了连贯性。

**2. 在设计用户认证功能时，后端API使用JWT（JSON Web Token）进行鉴权。前端Vue3应用在登录成功后，应该如何安全地存储JWT Token？**
A. 存储在 `localStorage` 中。
B. 存储在 `sessionStorage` 中。
C. 存储在 `vuex` 或 `pinia` 的状态管理器中。
D. 存储在 `HttpOnly` 的 `Cookie` 中（通过服务器设置）。

**正确答案：D**

**解析：** `localStorage` 和 `sessionStorage` 都存在XSS攻击风险，恶意脚本可以直接读取。`vuex/pinia` 中的状态在页面刷新后会丢失。`HttpOnly Cookie` 无法被JavaScript访问，能有效防御XSS攻击，是存储敏感Token的最佳实践。虽然选项A和B是常见做法，但从安全角度，D是最优解。

**3. 在实现响应式UI时，Vue3的 `<script setup>` 语法中，以下哪个选项能正确实现一个“发送消息”按钮的加载状态（当请求进行时禁用按钮并显示“发送中...”）？**
A. `const isLoading = ref(false);` 直接在模板中使用 `isLoading` 控制 `disabled` 属性。
B. `const isLoading = computed(() => { ... });` 依赖某个响应式数据计算状态。
C. `const isLoading = reactive({ value: false });` 在模板中使用 `isLoading.value`。
D. 使用 `v-if` 和 `v-else` 分别渲染按钮的不同状态。

**正确答案：A**

**解析：** 这是最直接且正确的做法。`ref(false)` 创建一个响应式布尔值，在发送请求前设置为 `true`，请求完成后设置为 `false`。模板中 `<button :disabled="isLoading">{{ isLoading ? '发送中...' : '发送' }}</button>` 即可完美实现。选项B的计算属性通常不用于直接控制异步状态；选项C虽然可行但更推荐使用 `ref`；选项D会导致DOM元素的销毁和重建，性能较差且不符合UI设计习惯。

### 二、简答题

**题目1：** 请描述在开发一个“智能学习助手”全栈应用时，用户从输入问题到获得答案的完整数据流。你需要明确指出以下关键环节：
*   **前端（Vue3）**：如何处理用户输入、管理对话状态、发送请求、渲染响应。
*   **后端（API）**：如何接收请求、处理鉴权、调用LLM API、持久化数据（如保存对话记录）、返回响应。
*   **LLM API**：请求和响应的格式。

**参考答案：**

1.  **用户输入与前端处理：**
    *   用户在Vue3组件的输入框中输入问题并点击“发送”。
    *   前端通过 `v-model` 获取输入内容，并立即将用户消息（`{ role: 'user', content: '...' }`）添加到 `messages` 这个响应式数组（对话状态）中，实现即时显示。
    *   前端设置一个 `isLoading` 状态为 `true`，禁用发送按钮并显示加载动画。
    *   前端将 `messages` 数组（包含历史对话）和用户ID（从JWT中解码或存储在Pinia中）作为请求体，通过 `axios` 或 `fetch` 向后端发送POST请求。

2.  **后端接收与鉴权：**
    *   后端（如Node.js + Express/FastAPI）接收到请求。
    *   中间件首先验证请求头中的JWT Token，确认用户身份。如果Token无效或过期，返回401错误。
    *   通过验证后，路由处理函数从请求体中提取 `messages` 数组和用户ID。

3.  **调用LLM API与持久化：**
    *   **调用LLM API：** 后端将接收到的 `messages` 数组（通常最后一条是用户最新问题）作为 `messages` 参数，调用LLM API（如OpenAI的 `/v1/chat/completions`）。API的请求体格式通常为 `{ "model": "gpt-4", "messages": [...] }`。
    *   **持久化对话：** 在发送请求给LLM之前或之后，后端将用户的问题和即将收到的AI回复（或流式传输的片段）保存到数据库中。通常的设计是：一个 `conversations` 表存储会话元数据（ID、用户ID、标题），一个 `messages` 表存储每条消息（会话ID、角色、内容、时间戳）。这样用户下次登录时可以加载历史对话。

4.  **返回响应与前端渲染：**
    *   **流式响应（推荐）：** 后端从LLM API获取流式响应（SSE），并实时地将生成的文本片段通过SSE或WebSocket推送给前端。前端每收到一个片段，就将其追加到 `messages` 数组中最后一条AI消息的 `content` 属性中，实现打字机效果。流式传输结束后，将 `isLoading` 状态设置为 `false`。
    *   **非流式响应：** 后端等待LLM API完整响应，然后将完整的AI回复内容存入数据库，并一次性返回给前端。前端接收到完整响应后，将其作为新消息添加到 `messages` 数组，并设置 `isLoading` 为 `false`。

**题目2：** 假设你的AI助手应用需要支持“多轮对话”和“历史会话管理”。请设计一个合理的数据库表结构（包含字段名、数据类型、主键、外键等），并解释如何通过后端API实现“加载特定历史会话”的功能。

**参考答案：**

**数据库表结构设计：**

1.  **`users` 表（用户表）**
    *   `id` (INT, PRIMARY KEY, AUTO_INCREMENT)
    *   `username` (VARCHAR, UNIQUE)
    *   `password_hash` (VARCHAR)  // 存储加密后的密码
    *   `created_at` (DATETIME)

2.  **`conversations` 表（会话表）**
    *   `id` (INT, PRIMARY KEY, AUTO_INCREMENT)
    *   `user_id` (INT, FOREIGN KEY REFERENCES `users(id)`)
    *   `title` (VARCHAR)  // 会话标题，可由LLM自动生成或用户自定义
    *   `created_at` (DATETIME)
    *   `updated_at` (DATETIME)

3.  **`messages` 表（消息表）**
    *   `id` (INT, PRIMARY KEY, AUTO_INCREMENT)
    *   `conversation_id` (INT, FOREIGN KEY REFERENCES `conversations(id)`)
    *   `role` (ENUM('user', 'assistant'))  // 消息角色
    *   `content` (TEXT)  // 消息内容
    *   `created_at` (DATETIME)
    *   `INDEX (conversation_id, created_at)`  // 为按会话查询和排序建立索引

**“加载特定历史会话”的API设计：**

*   **API端点：** `GET /api/conversations/{conversation_id}/messages`
*   **鉴权：** 请求头中需携带有效的JWT Token。后端验证Token后，获取当前用户ID。
*   **逻辑：**
    1.  后端收到请求，从URL参数中获取 `conversation_id`。
    2.  查询 `conversations` 表，确认 `conversation_id` 对应的 `user_id` 是否等于当前登录用户的ID，以确保用户只能查看自己的对话（权限校验）。
    3.  如果验证通过，查询 `messages` 表，`WHERE conversation_id = ? ORDER BY created_at ASC`。
    4.  将查询结果（一个包含多条消息的数组）格式化后作为JSON响应返回给前端。
    5.  前端接收到响应后，将消息数组赋值给 `messages` 响应式变量，并渲染在对话界面上。
